# 管理后台改进走查

更新时间：2026-07-10

本文档记录对当前雨城通 v2 管理后台的代码级走查结论。本轮未启动本地服务，主要依据 `apps/web/components/admin-poi-panel.tsx`、`/api/admin/map/poi-submissions`、POI 投稿工作流、仓储和既有需求文档评估。

## 1. 当前结论

管理后台已经能完成最小闭环：真实 `ldpass` 会话鉴权、读取投稿、审核通过/驳回、发布公开 POI，并通过事件总线发布 `PoiSubmitted`、`PoiReviewed`、`PoiPublished`。这足够验证“用户投稿 - 管理员审核 - 发布到公开地图”的通路。

POI 后台已经从调试列表推进到审核工作台第一版：能筛选状态/分类/关键词，展示分类名称和图标，展开详情，查看几何预览、图片、重复/相近提示，填写驳回理由，修正基础资料和点状坐标，并在发布前进行确认。后续的主要缺口已经转向地图审核工作台、线性/区域几何编辑、图片局部审核、父子 POI 关系、多语言资料和正式数据治理。

## 2. POI 后台主要缺口

### P0：审核效率与安全边界

- 状态筛选、分类筛选、关键词搜索和待办计数已有第一版；后续需要补批量操作、保存视图和更细的审核队列分组。
- 驳回理由已有弹窗和常用原因第一版，管理员可选择坐标偏离、分类不准、命名问题、信息不足、图片问题或疑似重复等模板后继续编辑；后续可将模板改为后台可配置。
- “通过”和“发布”仍是两个独立状态动作；发布按钮已增加确认弹窗，明确发布后会进入公开地图，并要求管理员勾选已核对坐标、分类、图片来源和重复提示。后续仍需要状态流转说明、正式发布预览和 PIN 二次确认。
- 分类名称、图标、是否允许公开投稿等上下文已有第一版展示；后续需要与多语言分类名和服务器 Profile 管理合并。
- 已有重复/相近地点提示和决策状态第一版：后台会读取公开地图标记快照，对同名、近距离、同分类附近的旧标记或已发布 POI 给出提示；管理员可以把单条提示标记为已忽略、待合并或重置判断，结果写入本地仓储并发布 `PoiConflictDecisionUpdated` 事件。真正的合并、替换、父地点推导和冲突审批仍待后续实现。
- 发布动作已有轻量高风险确认第一版；发布、归档、覆盖几何和批量操作后续仍应接入 PIN 二次确认。

### P1：地图与几何审核

- 后台已提供“地图查看”外链、代表坐标、点状 POI 的 X/Z 坐标修正第一版，以及只读审核地图预览第一版。当前点状坐标修正已从纯数字输入升级为“数字输入 + 小地图点选”第一版，会把当前点、原始点和附近真实参考标记放在同一个 SVG 坐标面里；审核地图预览会把投稿几何与附近真实标记、同分类标记、道路参考点和站点参考点放在同一个 SVG 小地图里，帮助管理员先判断大致位置关系。后续仍需要真正叠加瓦片、道路图和站点图层的地图选点控件。
- 点、线、矩形组、多边形都被 schema 支持，后台详情已有轻量 SVG 几何预览第一版，可辅助查看点序、线和面的大致形状；完整几何编辑器仍待后续实现。
- 代表坐标当前由几何包围盒中心推导。对线性对象或多矩形区域来说，这个点可能不在对象上，只能作为临时定位，不能作为审核依据。
- 缺少“在地图上选点/改点/补点”的后台控件。前台投稿已经支持地图选点，后台涉及坐标、站点、线路途径点和 POI 几何的字段也应复用同一能力。
- 缺少与现有标记点、道路、公共交通站点的叠加校验。管理员无法直接看出投稿是否落在道路、建筑、站点或父地点范围内。

### P1：投稿资料与图片审核

- 投稿人只显示 `ldpassUserId`，没有用户名、服务器账号验证快照、头像或历史投稿上下文，难以判断来源可信度。
- 图片已有单图缩略入口、本地上传图片基础审计信息和局部审核状态第一版：后台会展示存储文件名、MIME 类型、文件大小和文件更新时间；外部图片链接会明确提示无法读取本地元数据；管理员可以标记图片可用、图片不合格或重置审核状态，结果写入本地仓储并发布 `PoiSubmissionImageReviewed` 事件。当前仍不支持多图、图片说明、删除或替换。
- 外部图片链接和上传图片在审核体验上没有明显区分。后续应优先把公开展示图片落到可控存储，并记录审核通过状态。
- 简介、链接和图片没有拆分审核状态。现在 POI 通过时隐含全部资料通过，后续如果图片不合格但坐标和文字合格，会缺少局部处理能力。

### P2：数据维护能力

- 审核通过前已有低风险资料修正第一版：管理员可在待审核状态下修正标题、分类、简介和链接，并记录 `PoiSubmissionUpdated` 事件。图片、几何、父子关系和线性/区域结构仍需后续专用编辑器。
- 没有父子 POI、出入口、设施、楼栋、景点、代表 POI 等关联维护界面，只能依赖旧命名规则推导。
- 没有线性 POI/区域 POI 的专用流程。道路、线路、区域建筑和普通点状地点未来不应共用同一个极简卡片。
- 区域 POI 不应只依赖几何中心作为代表位置。大部分公园、学校、商业体、场馆等区域 POI 后续应由一个点状代表 POI 衍生，并通过子 POI 管理出入口、楼栋、景点和设施；只有行政区划等少量对象可能没有自然代表点。
- 地点名称、别名、简介和分类展示文案需要多语种支持。早期只考虑中文会让后续搜索、分享图、路线说明和地图详情的国际化返工变大。
- 道路 POI 后续可能需要高度、楼层或海拔层级信息，用于桥梁、地下通道、高架道路、站内通道和多层建筑的路线规划。
- 河流、湖泊、水域等区域 POI 目前缺位。它们不只是展示信息，也会影响步行路线，至少应作为“不可步行穿越”的约束候选，避免路径规划跨水域直连。
- 交通线路后台和 POI 后台存在明显重叠：公交、客运、部分轮渡接驳和地方铁路等线路需要站点、线路轨迹、道路吸附和方向信息；有些交通方式基本一定沿道路走行，适合复用 POI/地图审核工作台里的道路参考层和几何编辑能力。
- 没有发布后版本记录、回滚、归档和变更对比。正式公开数据需要能追踪“谁在什么时候改了什么”。
- 分类/Profile 管理已有第一版覆盖配置入口：管理员可以编辑分类名称、排序、可投稿开关、默认图标和一类多图标文件列表，并可上传或删除运行时分类图标；图标列表会在浅色、深色和地图感底色上做批量预览，辅助检查可读性。后续不同服务器语义色、排序、可投稿开关和图标文件本身都需要进入更完整的后台维护流程。

## 3. 建议改造阶段

### 阶段 A：把调试列表变成可用审核列表

优先级最高，改动相对低风险：

- 增加状态筛选、分类筛选、关键词搜索、待审核计数和只看待办开关。
- 读取 `/api/map/poi-categories`，在后台展示分类名称、图标和可投稿状态，而不是只显示 `categoryId`。
- 已处理第一版：驳回按钮改为弹窗，必须填写或选择驳回理由，并提供常用原因快捷填充。
- 已处理第一版：投稿卡片可展开详情，集中展示坐标、几何摘要、图片、链接、投稿人、审核记录和操作。
- 已处理第一版：发布按钮增加确认弹窗，明确“发布后将进入公开地图”，并要求管理员勾选已核对公开信息。

### 阶段 B：建立地图审核工作台

用于处理坐标、线性对象和区域对象：

- 在 POI 后台详情中嵌入可复用地图选点控件，支持定位投稿坐标、显示现有标记、显示道路/站点参考层。
- 点状 POI 支持管理员直接调整 X/Z 坐标；当前已在基础资料修正弹窗中提供第一版，后续需要抽出真正的后台地图选点控件。
- 线性 POI 支持点序预览、补点、删点、反转方向和从道路图吸附候选路线。
- 区域 POI 同时兼容多矩形和多边形输入，展示时转换为多边形；编辑时允许继续保留多矩形模式。
- 增强重复/冲突提示：当前已按同名、近距离、同分类附近给出第一版提醒，并支持管理员标记已忽略、待合并或重置；后续继续补与旧地图标记重合、父地点关系可推导、相似名称、真正合并队列和替换流程。
- 区域 POI 支持绑定代表点 POI；存在代表点时，后台优先展示代表点坐标和子 POI 关系，区域几何只作为范围边界。
- 道路和水域对象进入地图审核参考层。道路可选填高度/层级；水域用于提示路线规划约束和潜在不可通行区域。
- 交通线路后台复用地图审核工作台：线路站点、方向子线路、途径坐标、道路吸附和手工轨迹都在同一类地图编辑体验中处理。

### 阶段 C：进入正式数据治理

适合 POI 投稿量和管理员协作增长后推进：

- 公开 POI 变更采用版本草稿：创建、审核、发布、归档、回滚都保留记录。
- 引入管理员角色范围：内容、POI、交通数据、服务入口、票务可逐步拆分。
- 发布、归档、批量操作、覆盖旧公开数据等高风险动作接入 PIN 二次确认。
- 图片进入更完整的独立素材审核流，支持多图、替换、图片说明、对象存储和安全扫描；当前单图已有可用/不合格/重置状态。
- 后台提供分类、图标、语义色、排序、可投稿开关和服务器 Profile 管理。
- 分类管理支持一类多图标，管理员可以上传或登记图标文件、选择默认图标、调整图标归类，并预览图标在浅色、深色和地图底图上的可读性；当前已有第一版行内预览，后续再补批量素材库、可读性评分和真实底图抽样。
- 业务实体多语言进入后台维护：地点、区域、线路、站点和分类至少预留简体中文、繁体中文和英语字段，并与搜索别名、分享图和路线说明共用同一译名来源。

## 4. 事件驱动补充

当前已有：

```ts
interface PoiSubmittedPayload {
  poiId: string;
  title: string;
  categoryId: string;
  description?: string;
  href?: string;
  imageUrl?: string;
  geometry: MapGeometry;
}

interface PoiReviewedPayload {
  poiId: string;
  decision: 'approved' | 'rejected';
  reviewerId: string;
  reason?: string;
}

interface PoiPublishedPayload {
  poiId: string;
  categoryId: string;
  description?: string;
  href?: string;
  imageUrl?: string;
  geometry: MapGeometry;
  publishedAt: string;
}
```

后续建议补充：

```ts
interface PoiSubmissionUpdatedPayload {
  poiId: string;
  editorId: string;
  changedFields: Array<'title' | 'categoryId' | 'description' | 'href' | 'imageUrl' | 'geometry'>;
}

interface PoiDuplicateCandidateLinkedPayload {
  poiId: string;
  candidateMarkerId: string;
  linkerId: string;
  reason: 'same_name' | 'nearby' | 'manual';
}

interface PoiRelationUpdatedPayload {
  parentPoiId: string;
  childPoiId: string;
  relation: 'entrance' | 'exit' | 'facility' | 'building' | 'scenery' | 'nearby';
  editorId: string;
}

interface PoiAssetReviewedPayload {
  poiId: string;
  imageUrl: string;
  decision: 'approved' | 'rejected';
  reviewerId: string;
  reason?: string;
}
```

这些事件的目标是让审核、地图缓存刷新、搜索索引、通知投稿者、审计日志和未来数据库 Outbox 解耦。

## 5. 推荐下一步

阶段 A 第一版已基本完成，下一步建议转入阶段 B 的地图审核工作台，先做不会锁死未来数据模型的最小闭环：

1. 将只读审核地图预览升级为真正的内嵌地图预览，叠加瓦片、道路图、站点图层和现有公开 POI。
2. 将点状坐标修正的小地图点选第一版升级为可复用真实地图选点控件，并给后台线路/站点坐标复用。
3. 为线性和区域 POI 增加可视化几何查看与点序校验，先只读或半编辑，避免过早确定完整编辑器方案。
4. 把重复/相近提示的“待合并”决策升级为真正的合并队列、替换流程和发布前差异对比。
5. 继续补图片局部审核、父子 POI/代表 POI、多语言资料和 PIN 二次确认的事件与状态模型。

这一阶段的重点是让管理员能在同一个后台里判断“坐标是否真的落在正确位置”，同时继续保持事件驱动和未来数据库/Outbox 迁移的边界。
